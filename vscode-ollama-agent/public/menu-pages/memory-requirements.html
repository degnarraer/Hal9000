Bob's Memory Requirements

Purpose

Bob's Memory gives each signed-in user a private, inspectable memory layer that helps Bob answer the current message with useful context from prior conversations without replaying old chats or inventing profile details.

User Outcomes

  • Memory is saved per authenticated user and never by display name or email.
  • The Memory page shows short, medium, and long summaries, learned factoids, and recent chat messages.
  • The user can refresh memory, delete individual messages, delete individual factoids, and wipe their own memory after typing WIPE.
  • Clearing the visible chat does not delete saved memory.
  • Processed chat stays inspectable in the Memory page but leaves Bob's prompt context after it has been merged into short-term memory.
  • Bob uses memory as background context for the current message only.

Memory Horizons

Short Term
Merges chat messages since the last memory update into current task state, immediate goals, preferences stated today, and unresolved next steps.
Medium Term
Merges existing medium-term memory with existing short-term memory.
Long Term
Merges existing long-term memory with existing medium-term memory.

Update Process

  • Bob updates memory as a cascade: long term, then medium term, then short term.
  • The cascade runs after configured chat-message thresholds or when estimated context pressure reaches the model-context trigger.
  • Context pressure counts summaries plus unprocessed chat, not the full saved chat log.
  • New memory is not allowed to replace better existing memory just because it is newer.
  • Saved memory size is bounded by the configured model context tokens, prompt reserve, trigger ratio, and max words for each horizon.

Safety Rules

  • Durable factoids must be supported by user-authored transcript text.
  • Bob must not infer sensitive attributes, secrets, credentials, medical facts, or financial account data.
  • If memory storage or summarization is unavailable, chat should continue and the UI should show a clear memory error.
  • Assistant emotion metadata is saved with assistant messages and can guide Bob's visible state.

Architecture Contract

  • The memory store owns persistence, schema setup, user scoping, CRUD, and prompt assembly.
  • The memory skill service owns cascading memory merges, factoid extraction, transcript formatting, context-pressure estimation, scheduling intervals, memory budgets, and duplicate job suppression.
  • Express route handlers stay thin and return normalized JSON.
  • Model-generated memory artifacts pass deterministic filters before persistence.
